Krahnborn OS — what we just built and why it feels heavier than the prototype

The Python prototype was one file. The TypeScript version is seven files plus a database. The agent itself is identical. Everything else exists for one reason: the system needs to not lose context across triggers, pauses, and human responses.

1. The mental shift

The Python prototype and the TypeScript build do the same agent work. The brain — the LLM running inside the Claude Agent SDK with the Router system prompt — is unchanged. What's different is everything around the brain.

Python prototype

  • Run from CLI with a request file path.
  • Agent reads vault, runs sf org list, returns JSON.
  • Output goes to stdout.
  • Process exits. Nothing remembered.

TypeScript Phase 0

  • Triggered by an HTTP request (or CLI script).
  • A pipeline row is created in Postgres before the agent runs.
  • Same agent runs. Same JSON comes back.
  • The row is updated to completed with the result + session ID.
  • If the agent had paused for a question, the row would sit in awaiting_input until you answered, then resume.
The why in one sentence: the prototype proves an agent can do the work; the TypeScript build is the harness that lets the agent be triggered by anything (email, Slack, web form), survive pauses, and be picked up by a human mid-flight — none of which are possible with a one-shot script.

3. What lives in src/

src/ ├─ index.ts # Fastify boots, registers /health and /requests, listens on PORT ├─ routes/ │ └─ requests.ts # the POST /requests handler — the orchestrator ├─ agents/ │ ├─ router-agent.ts # system prompt + tool list + model — pure config, no execution │ └─ runner.ts # wraps Agent SDK query(), parses the final JSON ├─ db/ │ ├─ client.ts # Supabase client, single instance with service-role key │ └─ pipelines.ts # createPipeline / updatePipeline / logEvent ├─ cli/ │ └─ router.ts # run the agent without HTTP — `npm run router -- "..."` └─ config/ └─ env.ts # validates ANTHROPIC_API_KEY, SUPABASE_*, VAULT_PATH

Two layers worth distinguishing

  • Agent layer (agents/) — what the LLM does. Identical in spirit to the Python prototype. Prompt, allowed tools, model.
  • Operational layer (everything else) — how the agent gets triggered, persisted, observed, resumed. This is the part the prototype didn't have.

4. What gets stored

Two Supabase tables. Together they are the source of truth for "what's happening right now" and "what happened before."

pipelines — one row per request

ColumnTypeSet when
iduuidinsert
sourcetextinsert (e.g. "manual", "email")
raw_requesttextinsert (the user's words, verbatim)
client_idtextafter Router runs (e.g. "nri")
dev_hub_aliastextafter Router runs (sf CLI alias)
org_typetextafter Router (always "scratch" in Phase 0)
statustextstate machine: running → completed | failed | awaiting_*
session_idtextcaptured from the SDK so we can resume the agent later
current_stagetextwhich agent is in flight (Phase 0 only "router")

pipeline_events — append-only log

Every meaningful moment becomes a row: created, stage_started, stage_completed, stage_failed. The payload jsonb column carries whatever context that event needs (the routed JSON, error message, etc.). This is what gives us a paper trail when we ask "what did the agent do, why, and when?"

Why two tables? The pipelines row tells you the current state. The pipeline_events log tells you the history. Mutating one row would lose the timeline. Append-only events make audit + debugging trivial later.

5. Request flow, step by step

What actually happens when you curl POST /requests:

// src/routes/requests.ts (annotated)

const pipeline = await createPipeline({ source, raw_request });
// → INSERT INTO pipelines (source, raw_request, status='running', current_stage='router')

await logEvent({ pipeline_id: pipeline.id, event_type: 'created' });
await logEvent({ pipeline_id: pipeline.id, event_type: 'stage_started', stage: 'router' });

const routed = await runRouter(raw_request);
// → spawns Claude Agent SDK process, runs Sonnet with router prompt,
//   agent uses Read/Glob/Bash to inspect vault + sf org list,
//   final fenced JSON is parsed and returned (with session_id)

const updated = await updatePipeline(pipeline.id, {
  client_id: routed.client,
  dev_hub_alias: routed.recommended_dev_hub,
  org_type: 'scratch',
  session_id: routed.session_id,
  status: 'completed',
});

await logEvent({ pipeline_id: pipeline.id, event_type: 'stage_completed', stage: 'router', payload: routed });

return reply.send({ pipeline: updated, routed });

If runRouter throws, the catch path flips the row to status: "failed" and writes a stage_failed event with the error message. Either way, the row is never left in a stale running state.

6. Two ways to run it

HTTP (the real path)

npm run dev
# in another shell:
curl -X POST :3000/requests \
  -H "Content-Type: application/json" \
  -d '{ "source": "manual", "raw_request": "..." }'

Writes a row, runs the agent, returns the routed JSON. Use this when you're testing the full operational loop.

CLI (just the agent)

npm run router -- "add Account_Tier__c picklist to Lead for NRI"

# or from a file (matches the Python proto)
npm run router -- --file ./request.txt

Skips Supabase. Identical agent, but no row, no event, no HTTP. Use this when you want to debug the agent's reasoning in isolation.

8. What's next (so the scaffold makes sense)

Most of the indirection in Phase 0 only pays off in Phase 1+. Knowing what's coming makes the present feel less over-engineered:

  • More agent stages. Triage → Router → Work Identifier → Execution → Docs. Each is a new entry in src/agents/ with its own prompt and tools, invoked from routes/requests.ts in sequence.
  • Pause / resume. When an agent calls AskUserQuestion, the orchestrator catches it, sets status = 'awaiting_input', saves session_id, and ends the call. A Slack reply or web form later resumes the agent with query({ resume: session_id, prompt: answer }). This is why we capture session_id now.
  • Email / Slack triggers. Same POST /requests handler, called by a Gmail webhook or Slack event instead of a human curl. The pipeline doesn't care where the trigger came from.
  • Inbox view. A future GET /inbox endpoint reads pipelines sorted by status. The frontend (Phase 4) wraps that.
The takeaway: Phase 0's job was to prove the loop. The agent works. The DB persists. The HTTP entry exists. From here, every new capability is a new agent definition + a new field on the pipeline row + a new branch in the orchestrator. The shape of the system doesn't change — only what's wired into it.
